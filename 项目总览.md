# 华为任务型多轮对话项目一阶段

## 1. 项目背景与核心目标
本项目面向 IT 领域智能客服场景，目标不是构建一个单纯“会生成回复”的对话模型，而是构建一个基于知识库、具备多轮澄清与策略决策能力的智能客服系统。系统能够在知识库支持下自动回答用户问题，并在信息不足、证据不充分或问题不明确时，通过主动追问、澄清意图、补全关键信息等方式完成问题求解，而不是简单转接或拒答。
当前已有资源主要包括两部分：
第一，真实人工客服历史数据。这部分数据具有较强场景真实性，但存在明显问题：大量样本并未真正解决问题，而是进行了转接、模板化回复或中断式应答，不能直接作为高质量监督信号使用。
第二，已有成熟 RAG 系统。当前系统输入对话上下文，能够输出 Top-10 完整案例，为知识增强回答提供基础能力。后续项目不会立即重做底层 RAG，而是在现有 RAG 基础上继续做证据精筛选、信息压缩与状态信号建模。
因此，本项目的核心问题并不是“如何让模型生成一句像客服的话”，而是：
> 如何利用已有知识库与历史客服日志，恢复并学习“知识驱动下的正确交互策略”，使系统具备单轮回答、多轮追问、证据确认与最终回复的完整能力。

## 2. 项目总体思路
整体上，本项目采用“数据重建 + 证据加工 + 策略驱动 + 对话生成”的技术路线，构建一个由两大子系统协同组成的智能客服 runtime：
- 一条线负责对话能力与策略行为学习；
- 另一条线负责知识证据加工与状态信号提供；
两者最终在 runtime 层进行融合，形成完整的 agent 行为链路。
整个系统并不是让一个模型端到端同时完成检索、理解、决策和生成，而是通过模块解耦实现更强的可控性和可维护性。系统的核心执行流程如下：
1. 用户输入问题与历史对话，进行状态判断（判断1）
2. 现有 RAG 返回 Top-10 候选完整案例；
3. RAG 精筛选模块从 Top-10 中提取更适合当前轮使用的 reference 片段，并生成结构化状态信号；
4. 对话 runtime 基于历史上下文、reference 证据与状态信号，判断当前应执行的动作，如直接回答、先澄清、继续追问、拒答或转接；
5. SFT 微调后的对话模型根据当前策略与证据生成最终回复（判断2）；
6. 如未完成问题求解，则进入下一轮交互，重复上述过程。
因此，本项目最终要实现的并不是一个“静态问答系统”，而是一个具有知识感知能力和交互决策能力的多轮客服 agent。
## 3. 总体技术架构
整体技术架构可以抽象为四层。
### 3.1 数据层
数据层负责构建高质量训练数据与评估数据，包括：
- 原始人工客服日志
- 知识库与案例库
- RAG 输出的 Top-10 候选案例
- 重建后的策略监督数据
- 多轮追问与回答样本
- 无法回答/应拒答/应转接样本
该层的关键不是“原样使用人工客服数据”，而是对其进行任务重构：从原始行为中恢复“在知识库支持下，本应采取的合理动作”。
### 3.2 证据层
证据层基于现有 RAG 输出做进一步加工，主要功能包括：
- 候选案例重排序
- 最优 reference 片段定位
- 冗余内容压缩
- 候选之间一致性分析
- 状态信号抽取
该层的输出不只是文本片段，还包括供决策使用的结构化信号，例如：
- 当前证据是否充分
- 是否存在关键槽位缺失
- 候选证据是否一致
- 当前问题是否可答
- 是否建议追问
---
### 3.3 策略层
策略层负责决定系统当前应该做什么，而不是直接输出自然语言。其任务是根据：
- 当前轮用户输入
- 历史对话
- 证据层输出的 reference
- 结构化状态信号
判断当前动作，例如：
- DirectAnswer：直接回答
- AnswerWithConstraint：带条件回答/补充说明
- AskClarification：澄清用户意图
- AskMissingSlot：询问缺失关键信息
- CannotAnswer：当前无法回答
- Handoff：转人工/转平台
这一步是整个系统的核心，因为真正决定客服系统质量的，不是语言是否流畅，而是动作是否正确。
---
### 3.4 生成层
生成层基于策略结果完成自然语言输出。其职责不是独立思考，而是在明确动作约束下组织表达，例如：
- 若策略为直接回答，则根据 reference 给出准确答复；
- 若策略为追问，则生成最小必要追问语句；
- 若策略为拒答或转接，则给出清晰解释和下一步建议。
生成层对应的是经过 SFT 微调后的对话 runtime。
## 4. 数据构建方案
数据构建是本项目的第一核心。因为当前最大的风险不是模型能力不够，而是训练数据中充满错误策略和次优行为，必须先把数据修正成“可学”的形式。
## 4.1 原始数据来源
原始数据主要来自历史人工客服日志，初步分为以下几类：
1. 单轮已解决样本
    用户问题明确，人工客服在当前轮或极少轮次内完成回答。
2. 单轮转接样本
    人工客服未作实质性回答，而是直接转接、模板回复或中止处理。
3. 多轮已解决样本
    客服通过多轮追问、确认和引用知识完成问题处理。
4. 多轮未解决样本
    多轮交互后仍未解决，或最终转接、拒答。
---
## 4.2 数据重建原则
对于原始数据，不以“人工客服说了什么”为唯一标准，而以“在知识库支持下应如何处理”为重建目标。因此需要引入知识库与 RAG 输出重新判定样本属性。
重建后建议将数据划分为以下四种训练目标类型：
### Type A：可直接回答样本
当前输入信息已经足够，且知识证据明确，系统应直接给出回答。
### Type B：原始未答但知识可答样本
人工客服原本转接或未处理，但接入知识库后可完成回答。这类样本需要重写为“合理可答”的训练样本。
### Type C：需追问后可答样本
当前信息不足，但通过补充一个或多个关键槽位后可以继续检索并完成回答。
### Type D：确实无法回答样本
即使接入知识库，当前问题仍不在能力范围内，或必须依赖外部平台、人工身份、账户权限等信息处理，应输出拒答、转接或下一步建议。
---
## 4.3 数据标注目标
后续标注不能只停留在“能答/不能答”二分类，而要形成策略监督数据。建议每条样本至少标注以下内容：
- sample_type：A/B/C/D
- user_intent：用户当前真实意图
- knowledge_answerable：是否知识可答
- missing_slots：缺失哪些关键信息
- retrieval_needed：是否需要依赖知识库
- action_label：当前轮应执行的动作
- reference_span：最佳证据片段
- response_target：目标输出类型（回答/追问/拒答/转接）
- rewritten_response：重写后的目标回复
- multi_turn_path：如是多轮样本，标注其决策路径
其中最重要的是 action_label，因为这决定了后续策略学习的方向。
---
## 4.4 数据构建结果形式
最终训练集建议拆成两部分：
### 4.4.1 对话生成训练集
用于 SFT，学习在给定策略与 reference 下输出自然语言。输入可设计为：
- 历史对话
- 当前用户问题
- 当前动作标签
- reference 摘要
- 状态信号
输出为目标客服回复。
### 4.4.2 策略决策训练集
用于学习“当前轮应该做什么”。输入为：
- 历史对话
- 用户问题
- RAG 信号
- reference 信息
输出为动作标签与必要的追问目标。
---
## 5. RAG 侧增强方案
当前已有成熟 RAG，项目不以重构基础检索为优先目标，而是在其输出 Top-10 完整案例基础上做第二阶段增强。该部分是 zefeng weilai 的主要工作区域。
## 5.1 当前输入输出
现有 RAG：
- 输入：用户当前轮问题 + 历史上下文
- 输出：Top-10 完整案例
当前问题在于，Top-10 完整案例虽然能提供信息，但对 runtime 来说仍然过于粗糙，存在：
- 信息冗余
- 候选片段太长
- 多个案例之间相互冲突
- 不知道哪些片段最关键
- 无法直接支持“是否需要追问”的策略判断
因此需要增加一个RAG 后处理与证据增强模块。
---
## 5.2 证据增强模块目标
证据增强模块负责把 Top-10 完整案例转化为两类输出：
### 输出一：可供生成使用的 reference
包括：
- 最相关案例
- 最关键片段
- 必要上下文压缩结果
- 多片段融合后的证据摘要
### 输出二：可供策略使用的状态信号
例如：
- retrieval_confidence：检索置信度
- evidence_sufficiency：证据充分性
- evidence_consistency：证据一致性
- slot_missing：关键槽位是否缺失
- answerability_estimate：当前可答性估计
- need_followup：是否建议追问
- followup_target：建议追问哪一类信息
---
## 5.3 RAG 精筛选的关键任务
建议zefeng weilai的工作拆成以下几个方向：
### 任务1：Top-10 候选重排序
目标是从已有 Top-10 中识别最可能支撑当前问题的候选，必要时给出前 1~3 条高质量案例。
### 任务2：reference 片段定位
不是直接把整个案例喂给模型，而是定位真正有用的片段，减少噪声。
### 任务3：上下文压缩
对过长案例进行摘要或结构化压缩，使其更适合 runtime 使用。
### 任务4：状态信号抽取
从候选分布与 reference 内容中抽取结构化信号，供策略层使用。
---
## 6. Runtime 设计方案
Runtime 是整个系统最终对外工作的主体，它不是单一模型，而是一个策略驱动的执行过程。
## 6.1 运行时输入
每一轮 runtime 的输入包括：
- 历史对话上下文
- 当前用户问题
- RAG Top-10 候选
- 证据增强模块输出的 reference
- 状态信号
---
## 6.2 运行时动作空间
建议将 runtime 的动作空间明确定义如下：
- DirectAnswer
- AnswerWithExplanation
- AskClarification
- AskMissingSlot
- AskConfirmation
- CannotAnswer
- Handoff
动作空间越清楚，后续训练和评测越容易对齐。
---
## 6.3 运行时执行逻辑
### 第一阶段：回答前策略决策
根据历史上下文、reference 与状态信号，先决定当前动作。
例如：
- 若证据充分且问题明确，则直接回答；
- 若证据命中但缺少关键条件，则追问缺失信息；
- 若候选冲突明显，则先确认问题目标；
- 若知识库完全不支持，则拒答或转接。
### 第二阶段：自然语言生成
在动作明确的前提下，生成符合该动作的输出文本。
例如：
- DirectAnswer：直接给出步骤、原因或结论；
- AskMissingSlot：只问最关键的缺失项；
- CannotAnswer：说明限制并给出下一步建议。
### 第三阶段：多轮继续
若当前轮未闭环，则进入下一轮，并再次调用 RAG 与策略模块。
---
## 7. 模型训练方案
## 7.1 SFT 训练目标
SFT 的主要目标不是让模型“自由发挥”，而是让其在给定动作与证据约束下输出合理回复。因此建议输入模板中显式包含：
- 历史对话
- 当前用户问题
- 当前动作标签
- reference 摘要
- 状态信号摘要
输出为目标客服回复。
这样训练出来的模型更像“受控执行器”，而不是不受约束的聊天模型。
---
## 7.2 策略学习目标
策略学习可以先从监督学习开始，即先训练一个轻量决策模块或通过 prompt/classifier 预测当前动作标签。后续如果数据规模与效果允许，再考虑进一步强化。
初期不建议把策略学习做得过重，重点是先把动作标签体系和决策质量做扎实。
---
## 7.3 训练阶段建议
### Phase 1：基础对话能力对齐
先用已整理出的高质量可答样本与多轮追问样本做基础 SFT，让模型具备基本客服表达能力。
### Phase 2：引入 reference 与动作约束
在输入中加入 reference 与 action label，使模型学会“基于证据和动作生成”。
### Phase 3：多轮闭环训练
引入需追问后再回答的样本，训练模型在多轮过程中维持连贯性与任务推进能力。
---

## 9. 跨组协同接口
这个项目最容易出问题的地方，不是技术本身，而是两组最后接口对不上。所以建议一开始就把接口统一。
## 9.1 输出格式建议
RAG组的最终输出不要只是“一个压缩后的文本”，而要统一成结构化对象，例如：
```plaintext
{
  "references": [
    {
      "case_id": "xxx",
      "score": 0.91,
      "selected_span": "...",
      "compressed_reference": "..."
    }
  ],
  "signals": {
    "retrieval_confidence": "high",
    "evidence_sufficiency": "partial",
    "evidence_consistency": "consistent",
    "missing_slots": ["product_version"],
    "answerability_estimate": "need_followup",
    "followup_target": "product_version"
  }
}
```
这样对话组就能直接基于这些字段组织训练和 runtime 输入。
---
## 9.2 对话组 → RAG组 的反馈机制建议
对话组也需要反向给RAG组提供反馈，例如：
- 哪些 reference 太长，生成端不好用
- 哪些信号无助于动作判断
- 哪些缺失槽位定义不合理
- 哪些案例虽然高分但不支持实际回答
这样两条线才能形成闭环，而不是各自优化各自的指标。
## 10 . 最终目的
我们要做的不是一个只会检索后作答的客服模型，而是一个能够基于知识证据判断“现在该答、该问、还是该转”的多轮智能客服 agent。会话组负责让系统“会说、会对话、会按策略输出”，检索组负责让系统“拿到对的证据、看到足够的信号、支撑正确决策”，两条线最终在 runtime 层汇合。
目前需要的内容有：
会话组：
整体设计框架，数据处理pipeline的设计与标注体系的明确文档，最终实现对于已有数据的效果增强的标注数据（包括对话、知识和相应的策略标签，需要检验）以及SFT后接入RAG的模型性能。
检索组：
整体设计框架，对于检索结果的精简和信号的给定，从而有助于会话组的输出。

整体所有内容需要压实，能够系统化体系化的实现，比如：
1. 从低质量人工客服日志中恢复知识驱动的正确交互方法
2. 基于证据压缩与状态感知的高效RAG的策略控制
3. 基于两阶段的知识引导的多轮任务型混合式对话建模
